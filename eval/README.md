# 트레이너챗봇 의도 분류 측정

> 2026-07-23 측정. 캡스톤 회고에 "지금 다시 만든다면 성능 지표를 처음부터 측정하겠다"고
> 적어둔 것을, 사후에 실제로 측정한 기록.
>
> ⚠️ 캡스톤 당시 결과물이 아니다.

## 무엇을 재는가

트레이너챗봇(`dify/trainer-chat.yml`)은 사용자 질문을 **6개 유형으로 분류**한다:

| class | 유형 |
|---|---|
| 1 | 절약 방법·저축 팁 |
| 2 | 충동구매·소비 자제 |
| 3 | 앱·카드·요금제 추천 |
| 4 | 저축 목표 변경 |
| 5 | 예산·지출 분석 |
| 6 | **위와 무관한 모든 질문** (→ 거절 응답) |

배포된 Cloudflare Worker 게이트웨이를 호출해, 응답이 **거절 형태인가(class-6) / 정상 처리인가**를
판정한다. 정상 질문(class 1~5)을 거절해버리는 오분류가 사용자 피해가 가장 크므로 이를 중점적으로 본다.

## 측정 방식

- 골든셋 20문항 (`intent_golden.jsonl`) — 정상 14 + 무관 6
- LLM 응답이 실행마다 미세하게 달라지므로 **각 문항을 3회 호출해 다수결**로 판정
- 거절 응답은 고정된 패턴 문구("저축 트레이너라 그 질문엔 답하기 어려워요")로 식별

## 한계 (먼저 읽을 것)

1. **class 1~5 사이의 세부 오분류는 잡지 못한다.** 예를 들어 절약팁 질문이 예산분석으로
   잘못 분류돼도, 둘 다 "정상 처리"라 이 방식으로는 구분되지 않는다. 잡히는 것은
   **정상↔거절 경계의 오분류**뿐이다.
2. **게이트웨이 IP당 하루 40회 한도**(설계된 남용 방지 장치)에 걸려 전체 20문항 × 3회 = 60회를
   한 번에 완주하지 못했다. 아래 결과는 한도 소진 전까지 완료된 문항 + 별도 반복 확인한 문항이다.

## 확인된 결과

### 발견된 실제 결함 — 재현되는 오분류

**"충동구매를 어떻게 참지?"** (class-2로 분류돼야 함)를 3회 반복 호출한 결과 **3회 모두 거절**했다:

```
시도1: "저는 저축 트레이너라 그 질문엔 답하기 어려워요! 😅 하지만 절약 팁이나…"
시도2: "저는 저축 트레이너라 그 질문엔 답하기 어려워요! 😅 절약 팁이나 예산 관리에…"
시도3: "저는 저축 트레이너라 그 질문엔 답하기 어려워요! 😅 저축이나 절약에 관한…"
```

충동구매 상담은 분류기 정의상 class-2로 명시된 정상 기능인데, "충동구매를 어떻게 참지?"라는
직접적인 표현조차 무관 질문(class-6)으로 밀려난다. **노이즈가 아니라 재현되는 결함**이다.
반면 같은 의도의 "배달 자꾸 시켜먹는 습관 고치고 싶어"(S16)는 정상 처리됐다 —
표현에 따라 분류 경계가 흔들린다.

### 정상 처리·거절이 올바르게 갈린 문항

한도 소진 전까지 완료된 문항에서는 절약팁·앱추천·목표변경·예산분석·무관질문이 모두
기대대로 분류됐다. 다만 표본이 부분적이라 정확도 수치를 확정 지어 말하지 않는다.

## 개선 방향

- class-2(충동구매)의 트리거 표현이 좁다. "참다 / 유혹 / 자제" 같은 표현이 분류 지침이나
  few-shot 예시에 더 들어가야 한다.
- 정확도를 수치로 확정하려면 IP 한도에 걸리지 않는 경로(로컬 Dify 직접 호출 또는 한도 상향)로
  전체 골든셋을 여러 번 완주해야 한다. 현재는 "결함이 있다"까지만 확인됐다.
